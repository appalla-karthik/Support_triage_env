# Pivot Strategy

## Core recommendation

Do not abandon the current project. Instead, pivot it upward.

The smartest move is to keep the support triage foundation and evolve it into a
more ambitious environment:

> **Support Operations Control Tower**

This keeps your existing work relevant while giving the judges a much bigger and
more original story.

## Why this pivot is strong

Many teams will build generic assistants, meeting planners, or thin wrappers around
existing tasks. Your current problem is stronger because it already has:

- enterprise realism
- measurable reward logic
- safety constraints
- meaningful failure modes
- room for agent training

The pivot should make those strengths more visible, not replace them.

## Best unique angle

The most unique version of this project is:

### A support command center with policy drift, specialist handoffs, and delayed outcomes

That means moving from "single agent solves a ticket" to:

- the queue changes over time
- internal teams react with delays
- the quality of escalation notes matters later
- policies can change between episodes
- the agent must optimize business outcomes, not just immediate correctness

## Recommended pivot path

### Stage 1. Strengthen the current single-agent world

Add more realism without changing the basic interface:

- larger mixed queues
- SLA timers
- enterprise account weighting
- reopen events after poor resolution
- richer task families such as account access, fraud, migration issues, and VIP escalations

This deepens Theme 3.1 and Theme 2 right away.

### Stage 2. Add multi-actor behavior

Introduce explicit internal actors:

- engineering triage
- billing specialist
- trust and safety reviewer
- customer simulator
- support manager or oversight reviewer

Now the environment becomes much more interesting because the agent must model
others' behavior and incentives. This pushes the project toward Theme 1 without
changing the business setting.

### Stage 3. Add drift and delayed consequences

This is where the project can become genuinely memorable:

- routing schemas change
- policy text changes
- API contracts for internal tools shift
- escalation templates evolve
- downstream outcomes arrive several steps later

That makes the environment feel closer to a real enterprise workflow.

## Best theme combination after pivot

If you implement the pivot well, the final project can credibly claim:

- **Theme 3.1** as the foundation
- **Theme 2** through long support sessions and delayed outcomes
- **Theme 1** through specialist coordination and oversight

That hybrid is stronger than trying to force a pure single-theme story.

## Concrete features that would improve your chances

### 1. Queue-level objectives

Move beyond per-ticket scoring and add:

- weighted backlog risk
- SLA breach penalties
- enterprise account retention value
- unresolved security exposure

### 2. Tool-like internal systems

Add actions such as:

- create escalation packet
- query account history
- open incident
- check billing ledger
- request engineering reproduction checklist

This makes the environment feel more like real work.

### 3. Policy and schema drift

This is a very strong differentiator because it tests robustness rather than simple
memorization. Examples:

- refund timeline rules change
- resolution codes change
- engineering requires a different escalation schema
- security recovery wording is updated

### 4. Oversight layer

Add an oversight or QA agent that:

- reviews the frontline agent's decisions
- flags unsafe communication
- explains failures
- adds bonus reward for well-justified actions

This creates a strong bridge to the multi-agent theme and to scalable oversight.

### 5. Before versus after training evidence

Show measurable improvement on:

- final score
- success rate
- policy violation rate
- average steps to resolution
- queue-ordering correctness

## What not to do

Avoid pivots that discard your strongest assets:

- do not reframe it as a generic chatbot benchmark
- do not over-promise multi-agent complexity if you only have one active agent
- do not pitch self-improvement as the primary theme unless you actually build adaptive curricula
- do not make the environment huge if the reward logic becomes vague

## Best concise pitch after pivot

> We built an enterprise support operations environment where an agent must manage
> a live queue, communicate safely with customers, route issues across specialist
> teams, adapt to policy drift, and optimize delayed business outcomes.

That sentence is specific, grounded, and noticeably stronger than "support ticket
classifier."
